# Result Sink — Design

**Date:** 2026-05-25
**Status:** Draft. Contract is locked (it's an upstream dependency of obacloud
PR #747); the validator-side implementation is open.

## Purpose

Let a caller — specifically obacloud's `ServerValidationJob` — read the
validator's report back from a Postgres database after a Render one-off job
finishes, instead of scraping it from stdout. The validator continues to print
to stdout exactly as today; the database write is **purely additive** and only
activates when the caller supplies result-sink fields in the input config.

## Why

obacloud invokes the validator as a Render one-off job using the same pattern
as its existing `api-key-service`: base64-encoded JSON payload as the start
command, `/app/entrypoint.sh` decodes it. The Render job object exposes status
but not stdout, so the caller needs a side-channel to retrieve the structured
report. The api-key-service uses a results table in the org's OBA Postgres; the
validator should match that pattern so obacloud can reuse its existing
`ObaDatabase::FetchResult` reader (extended with a `result_table:` parameter and
allow-list).

Full upstream context: `OneBusAway/obacloud#747`, spec
`docs/superpowers/specs/2026-05-25-oba-server-validator-integration-design.md`
in that repo (§Validator-side change). The change documented here matches what
obacloud is built against.

## The contract

### Input — new optional fields in the base64 payload

The fields are **invocation inputs**, not part of the report contract. They do
NOT appear in `schema/oba-validator-report.schema.json`. They live alongside
the existing validator config (`obaServerURL`, `apiKey`, `dataSources`) in the
same top-level JSON object that `entrypoint.sh` decodes.

| Field | Type | Notes |
|---|---|---|
| `db_url` | string | JDBC-style URL, e.g. `jdbc:postgresql://host:5432/dbname`. Mirrors what obacloud's `ApiKeyServiceJob` sends. Treat as the activation flag for the sink: **if absent or blank, the sink is disabled and the validator behaves exactly as today**. |
| `db_user` | string | DB user (the org's internal Postgres role). |
| `db_pass` | string | DB password. |
| `correlation_id` | string | UUID generated by the caller. Used as the row key. |
| `result_table` | string | Table name, e.g. `"oba_validator_results"`. Caller controls this. The validator MUST NOT interpolate it into SQL without explicit allow-listing — see Security below. |

All five must be present and non-blank together. If any are missing while
`db_url` is set, fail with `status: "error"` and a clear `error_message`
explaining which field is missing (and don't attempt to write).

### Output — one row in `result_table`

After the normal validator run (success or failure), if the sink is configured,
write **exactly one row**, creating the table on first write with
`CREATE TABLE IF NOT EXISTS`:

```sql
CREATE TABLE IF NOT EXISTS <result_table> (
  correlation_id TEXT PRIMARY KEY,
  status         TEXT NOT NULL,
  result_data    TEXT,
  error_message  TEXT
);
```

Then:

```sql
INSERT INTO <result_table> (correlation_id, status, result_data, error_message)
VALUES ($1, $2, $3, $4)
ON CONFLICT (correlation_id) DO NOTHING;
```

Column shape is fixed by obacloud's reader (`ObaDatabase::FetchResult`):
`correlation_id`, `status`, `result_data`, `error_message`. Don't add columns.

### The `status` column vocabulary — read this carefully

This is the single subtlest part of the contract. `status` is the **retrieval
status**, not the validation **verdict**. The two are different and conflating
them silently throws away FAIL reports.

obacloud's reader collapses `row.status == "failed"` into its own
`success?: false` and **discards `result_data`**. If the validator naively
wrote `status: "failed"` when its exit code is 1 (found failures), obacloud
would throw away a perfectly good FAIL *report* and surface a confusing "no
report" failure to the operator.

So `status` is one of exactly two values:

- **`"completed"`** — the validator produced a report (PASS *or* FAIL verdict).
  `result_data` is the full report JSON document (the same payload
  `--json` prints to stdout, marshalled as a string for the TEXT column).
  `error_message` is empty. Exit code 0 or 1.

- **`"error"`** — the validator could not produce a report (config/usage
  error — the `errorDocument` schema variant). `result_data` is empty.
  `error_message` carries the cause. Exit code 2.

**The verdict lives inside `result_data` at `summary.verdict`** (already part
of the published report schema, values `"PASS" | "FAIL"`). obacloud parses it
from there, not from this column. A FAIL verdict therefore arrives as
`status: "completed"`, `result_data: <report JSON with summary.verdict =
"FAIL">`. That's correct.

### Stdout and exit code — unchanged

The validator **still prints to stdout** (the report document on completed,
the `errorDocument` on error) and **still exits with its existing exit
codes** (`0` no failures, `1` ≥1 failure, `2` config/usage error). The DB
write is purely additive — every existing consumer that reads stdout or
checks the exit code is unaffected.

If the DB write itself fails after a successful validation (network blip,
table-creation race, etc.), the validator should:
1. Log the DB error to stderr.
2. Still print the report to stdout (already done before the write attempt
   in the recommended ordering below).
3. Still exit with the *validation* exit code (0/1), not a DB-related code.
   The caller's polling/timeout path is the safety net — if the row is
   missing, obacloud times out after 15 minutes with a clean error.

### Recommended ordering inside `main`

1. Parse the input config (including the optional sink fields).
2. Run the validator as today.
3. Print the report (or `errorDocument`) to stdout — same as today.
4. **Then**, if the sink is configured, write the row. This ordering means
   a DB-write failure can't prevent obacloud from seeing the result via the
   alternate path (Render logs), and stdout consumers are never affected.
5. Exit with the validator's exit code as today.

## Backward compatibility

The change is purely additive:
- Stdout output: unchanged.
- Exit codes: unchanged (still `0`/`1`/`2`).
- The published `oba-validator-report.schema.json`: unchanged.
- CLI flags: unchanged.
- Behavior when `db_url` is absent from the payload: **identical to today**.

Existing consumers (local `docker run`, the README's documented `--json`
workflow, any other caller reading stdout) are not affected.

## Security

- API key (existing) and DB creds (new) appear **only inside the base64
  blob**. Never in argv, never in process listings, never in logs.
  `entrypoint.sh` already handles this for the existing `apiKey`; the new
  fields ride the same channel.
- `result_table` is caller-controlled. Treat the table name as untrusted
  input and **enforce an allow-list before interpolating it into SQL**.
  Mirror obacloud's `ObaDatabase::FetchResult` allow-list (`%w[api_key_results
  oba_validator_results]`) — for the validator side, the allow-list is just
  `["oba_validator_results"]` since the validator only ever writes to its
  own table. Reject anything else with `status: "error"`,
  `error_message: "unsupported result_table"`.
- Existing API-key redaction in the validator's `errorDocument` and report
  output continues to apply (`validator/util.go#redact`). DB-creds redaction
  in errors should follow the same pattern: never echo `db_pass` in any
  log line or `error_message` value.
- TLS to Postgres: use `sslmode=require` by default. obacloud uses
  Render's internal connection string (private network), so the requirement
  is mild — but `sslmode=require` is the safer default.

## Architectural recommendations

The implementer can choose, but here's a sensible default shape:

### New package: `sink/`

Add a small package, e.g. `sink/sink.go`:

```go
package sink

type Config struct {
    DBURL          string `json:"db_url"`
    DBUser         string `json:"db_user"`
    DBPass         string `json:"db_pass"`
    CorrelationID  string `json:"correlation_id"`
    ResultTable    string `json:"result_table"`
}

func (c Config) Configured() bool { return c.DBURL != "" }

func (c Config) Write(ctx context.Context, status, resultData, errorMessage string) error {
    // 1. Allow-list result_table.
    // 2. Open a pgxpool / pgx.Conn against c.DBURL with the creds.
    // 3. CREATE TABLE IF NOT EXISTS.
    // 4. INSERT with parameterized values (correlation_id, status, result_data, error_message).
    // 5. Close.
}
```

### Config wiring

`config.Config` already holds the validator inputs. Add the five new fields
(`config/config.go`). They're optional — zero-value means "no sink." Don't
print them in `--help` or in error messages (they're invocation inputs, not
user-facing knobs).

### Main wiring

After the existing report-printing in `cmd/oba-validator/main.go`, gate on
`cfg.Sink.Configured()`:

```go
if cfg.Sink.Configured() {
    status, errMsg := "completed", ""
    if err := cfg.Sink.Write(ctx, status, reportJSON, errMsg); err != nil {
        fmt.Fprintf(os.Stderr, "result sink write failed: %v\n", err)
    }
}
```

For the `errorDocument` branch (validation couldn't produce a report), write
with `status: "error"` and the error message.

### DB driver choice

**Recommend `pgx/v5`** (`github.com/jackc/pgx/v5`) over `database/sql` +
`lib/pq`. Reasons: maintained, native PostgreSQL types, simpler ergonomics
for one-off connections, smaller surface than the stdlib `sql` indirection.
The validator already has a small dependency footprint; adding pgx is
acceptable for the value.

If the implementer prefers `database/sql` for stdlib consistency, that's
fine — same row shape, same allow-list discipline.

### Connection URL parsing

`db_url` arrives in JDBC form (`jdbc:postgresql://host:5432/dbname`) because
that's how obacloud's existing `ApiKeyServiceJob` formats it. The validator
must strip the `jdbc:` prefix before handing to pgx, then thread the user
and password via a `pgx.ConnConfig` (or assemble a DSN). Don't naively pass
the JDBC URL to pgx — it'll fail to parse.

## Testing

- **Unit:** `sink_test.go` against an in-memory `pgxmock` or a `testcontainers`-
  spun Postgres. Cover:
  - `Configured()` true/false matrix.
  - `result_table` allow-list rejects unknown names.
  - `CREATE TABLE IF NOT EXISTS` is idempotent across two writes.
  - `INSERT … ON CONFLICT DO NOTHING` works if a writer retries.
  - `Write` propagates connection errors to the caller (so main can log to
    stderr).
- **Integration:** an env-gated test like the existing
  `OBA_VALIDATOR_LIVE` integration. New env: `OBA_VALIDATOR_DB_DSN`. Spin a
  local Postgres, run the validator with a complete sink config, assert
  the row landed.
- **`config_test.go`:** new cases for the parsed optional fields and for
  the "db_url set but a sibling is missing" error path.

## Deployment

The new image must be deployed to the existing Render service
`crn-d8a9p20jo6nc73e9h9jg` before obacloud's PR #747 is usable end-to-end.
**During the deploy gap**, obacloud's failure mode is benign: the validator
runs and prints to stdout (which Render accepts as success), no row is
written, and obacloud's polling job times out after ~15 minutes with a
"No result found for correlation_id" error message. No data loss, no
silent corruption.

The image tag, `render.yaml`, and the Render service id are unchanged.

## Open questions

- **pgx vs database/sql.** Recommendation above (pgx), but implementer
  preference is fine — the contract is the same.
- **Connection pool vs single connection.** A single `pgx.Conn` for one
  INSERT is simplest. A pool buys nothing here since the validator does
  exactly one write per run.
- **`ON CONFLICT DO NOTHING` vs `DO UPDATE`.** `DO NOTHING` matches the
  api-key-service pattern and keeps the contract idempotent under retry
  without overwriting earlier writes. Recommended unless obacloud
  explicitly requests upsert semantics later.
- **Connection timeout.** Default to 5s connect + 5s statement timeout,
  mirroring obacloud's `ObaDatabase::FetchResult` (`SET statement_timeout
  = '5s'`). Anything longer makes the validator hang on bad creds.
